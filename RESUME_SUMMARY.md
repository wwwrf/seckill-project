# 高并发秒杀系统（简历浓缩版）

项目简介：基于 Gin 的秒杀系统，重点解决高并发下的库存一致性、消息可靠投递和认证限流治理，覆盖注册登录、秒杀下单、异步建单、轮询、支付、超时取消全链路。

1. 我采用 Redis Lua 原子预扣（`DECR + SADD + HSET pending`）替代分布式锁串行，核心原因是减少网络往返和锁竞争；再用 MySQL `uk_user_activity` 与 `available_stock > 0` 条件更新兜底，形成“内存快裁决 + 数据库终态约束”的一致性方案。

2. 我将主消息链路重构为 Outbox 状态机（`seckill_tx_tasks`）：请求只负责写 Outbox，后台调度器同步投递 MQ，失败重试并支持 dead 自动重放；消费端通过唯一键冲突幂等 ACK，系统语义为 at-least-once 但可追踪、可恢复。

3. 我补齐超时取消可靠性：建单后发送 RocketMQ 30 分钟延迟消息，发送失败写 `order_timeout_tasks` 由补偿器重投；配合 `canceled/processing` 标记解决 Cron 回滚与 MQ 迟到消息赛跑，避免误回滚和脏建单。

4. 我把安全治理落到可复核实现：认证为 bcrypt + JWT，登录对不存在用户做 `dummyHash` 比对抗时序枚举；trace 为 `X-Trace-ID` 透传（雪花 ID），限流为应用层令牌桶（用户级 + 秒杀接口 IP 前置限流），并固定中间件顺序 `Trace -> JWT -> RateLimit`。

5. 我输出了可复核压测数据而非抽象方案：读链路单机 `9998 RPS`、`HTTP P95=1.95ms`、错误率 `0%`；交易链路稳定档 `13.30 TPS`（`txn P95=1223ms`）和峰值档 `23.55 TPS`（`txn P95=8103ms`），并将容量结论拆分为“稳定容量”和“极限容量”。

6. 我给出量化优化结果：热点读吞吐 `1199.99 -> 9998.08 RPS`（约 `+733%`），全链路稳定 TPS `3.15 -> 13.30`（约 `+322%`），峰值 TPS 提升至 `23.55`，全链路 HTTP 错误率 `4.74% -> 0%`；CPU/内存因当前未沉淀可复核水位样本，简历中不写不可复现数字。
