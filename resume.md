# 高并发秒杀系统（简历改写版）

项目简介：基于 Gin 的秒杀交易系统，围绕“库存一致性、异步可靠投递、认证与限流治理”三条主线做工程化落地，覆盖注册登录、秒杀下单、异步建单、结果轮询、支付、超时取消完整闭环；技术栈为 Go + Gin + MySQL + Redis + RocketMQ + JWT + k6。

1. 我主导重构了秒杀一致性主链路，采用 Redis Lua 一次性完成 `DECR库存 + SADD已购 + HSET pending` 原子预扣，而不是分布式锁串行方案；核心决策是把多步网络往返压缩为单次 EVAL，减少锁竞争与锁续期复杂度，同时在 DB 侧用 `orders.uk_user_activity(user_id,activity_id)` 和 `seckill_activities.available_stock > 0` 条件更新兜底，形成“Redis 快速裁决 + MySQL 最终约束”的双保险，压测与回归中未出现超卖样本。

2. 为解决“请求成功后主消息投递失败，导致 Redis 预扣与 DB 建单状态不一致且不可追踪”的问题，我将主消息链路改造成可审计状态机：`seckill_tx_tasks` 持久化 + 1 秒调度器批量同步投递 + 指数退避重试 + dead 自动重放；消费端以唯一索引冲突做幂等 ACK。这个方案不承诺 exactly-once，但把黑盒风险转成 at-least-once 下的“可重试、可追踪、可回放”，显著降低人工排障成本。

3. 我补齐了超时取消与补偿赛跑治理：建单后发送 RocketMQ 30 分钟延迟消息（DelayLevel=16），若发送失败落 `order_timeout_tasks` 由补偿器重投；同时引入 `canceled/processing` 原子标记，解决 Cron 回滚与 MQ 迟到消息并发冲突，避免误回滚和脏建单；超时取消路径按“先 DB 状态流转并回补库存，再 Redis 回滚库存/已购标记”执行，保证最终一致性收敛。

4. 我把安全与治理从“口号”落实到代码细节：认证采用 bcrypt + JWT，登录对不存在用户执行 `dummyHash` 比对降低时序枚举风险；链路追踪使用自定义 `X-Trace-ID` 透传（雪花 ID 生成），限流采用应用层令牌桶（`x/time/rate`），粒度为“用户级限流 + 秒杀接口 IP 前置限流”，并明确中间件顺序 `Trace -> JWT -> UserRateLimit`，在 NAT 场景下减少误伤。

5. 我重构了压测口径并沉淀了可复核数据：读链路在单机本地达到 `9998 RPS`，`HTTP P95=1.95ms`，错误率 `0%`；全链路交易在稳定档（L1）达到 `13.30 TPS`（`txn P95=1223ms`，HTTP 错误率 `0%`），峰值档（L2）可到 `23.55 TPS`，但 `txn P95` 上升到 `8103ms`，据此将容量结论拆分为“稳定容量”和“极限容量”，避免只报峰值。

6. 我给出并验证了优化收益而非只写方案：相较前一轮基线，热点读吞吐从 `1199.99 RPS` 提升到 `9998.08 RPS`（约 `+733%`），全链路稳定 TPS 从 `3.15` 提升到 `13.30`（约 `+322%`），峰值 TPS 提升到 `23.55`；同时把全链路 HTTP 错误率从 `4.74%` 降到 `0%`。CPU/内存方面，仓库当前仅完成 Prometheus/Grafana 监控脚手架，尚未沉淀可复核的固定水位样本，因此简历只保留已落盘验证的数据，不写不可复现的数字。

# 高并发秒杀系统（简历精简重写版）

项目简介：基于 Gin 的高并发秒杀系统，核心解决三件事：高并发下不超卖、一人一单幂等、消息链路可恢复；覆盖注册登录、秒杀下单、异步建单、结果轮询、支付、超时取消闭环。

1. 我在一致性主链路上采用 Redis Lua 原子预扣（库存扣减 + 已购标记 + pending 记录同脚本执行），并在 MySQL 侧用 `uk_user_activity(user_id,activity_id)` 与 `available_stock > 0` 条件更新兜底；技术选型重点是把高并发裁决放在内存侧完成，同时保留数据库终态约束，避免超卖和重复下单。

2. 为解决“主消息发送失败会造成预扣成功但订单未落库，且失败任务难以追踪”的问题，我将主消息改造为 Outbox 可靠投递链路：`seckill_tx_tasks` 持久化 + 1 秒调度器批量同步投递 + 指数退避重试 + dead 自动重放；消费端通过唯一键冲突幂等 ACK，系统语义为 at-least-once，但具备可追踪、可回放、可恢复能力。

3. 我补齐超时取消与补偿治理：建单后发送 RocketMQ 30 分钟延迟消息（DelayLevel=16），发送失败写入 `order_timeout_tasks` 由补偿器重投；同时引入 `canceled/processing` 原子标记规避 Cron 回滚与 MQ 迟到消息并发竞态，保证最终一致性收敛。

4. 我将安全和治理落到可复核实现：认证为 bcrypt + JWT，登录对不存在用户执行 `dummyHash` 抵御时序枚举；链路追踪使用 `X-Trace-ID` 透传（雪花 ID 生成），限流采用应用层令牌桶（用户级限流 + 秒杀接口 IP 前置限流），中间件顺序固定为 `Trace -> JWT -> RateLimit`。

5. 压测数据（单机本地，i7-14650HX）显示：读链路达到 `9998.08 RPS`，`HTTP P95=1.95ms`，错误率 `0%`；全链路交易在 `3082~3645 req/s` 请求压力下，成交能力为 `127.29~141.61 TPS`，HTTP 处理 `P95=8.35~8.88ms`、错误率 `0%`，业务闭环 `P95` 约 `8.9~9.1s`（由异步建单 + 轮询确认模型决定）；MQ 侧观测到短时积压峰值后可回落，且未出现重试堆积与死信主题。

6. 我给出可量化优化结果：热点读吞吐从 `1199.99 RPS` 提升到 `9998.08 RPS`（约 `+733%`），全链路交易从旧口径个位数 TPS 提升到稳定百级 TPS；同时将容量口径拆分为“接口承压能力”和“业务成交能力”，避免只报峰值造成误导。CPU/内存暂不写具体数字，因为当前仓库没有可复核的固定水位样本。
