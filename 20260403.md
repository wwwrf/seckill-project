### 1.3 推荐改写版本（可直接放简历）

#### 项目简介（优化版）
基于 Gin 的高并发电商秒杀系统，围绕“认证安全、库存一致性、异步可靠投递”三类核心问题进行设计与落地。系统采用 Redis Lua 原子预扣、多级缓存、RocketMQ 异步削峰、Outbox 可靠消息与补偿任务机制，完成用户注册/登录、商品查询、秒杀下单、异步建单、订单支付、超时取消闭环，并通过分层压测与消息队列指标观测验证系统承压与收敛能力。

#### 主要内容（优化版）
1. 秒杀一致性主链路
- 设计 Redis Lua 原子预扣（库存扣减 + 已购标记 + pending 记录同脚本执行），结合 MySQL 唯一索引与条件更新兜底，保障高并发下不超卖、不重复下单。
- 落地 canceled/processing 并发标记机制，解决 Cron 补偿与 MQ 消费竞态，避免迟到消息导致脏建单。

2. 认证与安全治理
- 实现 bcrypt 密码哈希、JWT 鉴权、用户级限流与 IP 前置限流，并统一接入中间件链路。
- 登录流程对不存在用户执行 dummyHash 比对，降低时序攻击风险；统一接入 trace、鉴权、限流提升可观测性与可治理性。

3. 异步建单与超时取消
- 实现主消息 Outbox（seckill_tx_tasks）+ 调度器同步投递 + dead 自动重放，提升消息可追踪与可恢复能力。
- 建单成功后发送 30 分钟延迟消息，消费端对未支付订单执行 DB 状态流转与 Redis 库存回滚，保证最终一致性。

4. 测试与容量评估（建议新增到简历）
- 采用分段加压方法验证容量拐点：读链路分 8 档（1.2k→10k RPS）；交易链路 L1/L2/L3 使用 k6 `ramping-arrival-rate` 模型（固定目标到达率，逐阶段抬升压力），用于识别系统从稳定区到退化区的转折点。
- 交易链路在三档压力下均达到百级 TPS（L1/L2/L3=127.29/127.59/141.61），对应 P95 分别为 9075/9136/8891ms，具备稳定成交能力；在更高注压下主要表现为延迟升高与压测端 `dropped_iterations` 增加。
- 消息队列观测指标：`tx_diff_max` L1/L2/L3=200/8/8，`timeout_diff_max`=0/21/15；未观测到重试堆积与死信主题。

### 1.4 面试时建议口径（简短）
- 不只追求高 RPS，而是同时保证正确性（不超卖）、可靠性（消息可恢复）、可观测性（可量化定位瓶颈）。
- 高并发下“快速拒绝（售罄/限流）”是系统保护策略，不等于系统崩溃。

## 二、RocketMQ 9 个问题（结合项目实现）

## 2.1 如何保证消息队列的高可用？

### 项目里怎么做
- 业务层采用 Outbox：
  - 主消息：`seckill_tx_tasks` + `TxMessageDispatcher` + `TxTaskReplayer`
  - 超时消息：`order_timeout_tasks` + `TimeoutMessageCompensator` + `DeadTaskReplayer`
- 即使 MQ 短时不可用，任务仍在 DB，后续可重试/重放。

### 还能怎么做
- RocketMQ 集群层面：NameServer 多节点、Broker 主从（DLedger/主从复制）、跨可用区部署。
- 客户端层面：多 NameServer 地址、超时与重试参数分级配置、熔断降级。

### 为什么当前这样选
- 该项目重点先解决“应用侧可靠交付”和“可恢复性”。
- 在单机开发环境下，Outbox 能把不可见丢失变成可观测可补偿，工程收益最高。

---

## 2.2 如何保证消息的顺序性？

### 项目里现状
- 当前消费者使用并发消费（Clustering + 多 goroutine），并未强制全局有序。
- 业务上通过幂等和状态机（pending/canceled/processing）保证最终正确，而非依赖严格顺序。

### 可选方法
- 局部顺序：按订单号/用户ID选择 queue（sharding key），同 key 进入同一 queue，并使用顺序消费。
- 全局顺序：单队列单消费者（吞吐代价高，一般不推荐）。

### 选型说明
- 秒杀场景更重吞吐和最终一致性，通常采用“并发消费 + 幂等”而不是“强顺序”。

---

## 2.3 如何避免消息重复消费？

### 项目里怎么做
- DB 唯一约束（`uk_user_activity`）作为最终幂等防线。
- 消费端遇到重复下单（Duplicate Key）按“已处理”ACK，不重复建单。
- Redis 已购标记 + DB 唯一约束双重防重。

### 可选方法
- 业务去重表（message_id + 状态机）。
- 幂等 token（请求侧）。
- 状态机版本号/乐观锁。

### 选型说明
- DB 唯一约束实现简单且强一致，适合订单类幂等落地。

---

## 2.4 大量消息在 MQ 长时间积压，如何解决？

### 项目里已有能力
- 已做 lag/inflight 指标观测。
- 有 Outbox 重试与 dead 重放机制，可避免发送侧静默失败。

### 实战处理手段
1. 先止血：临时扩容消费者并发、提高消费批量、必要时降低生产侧速率（限流/降级）。
2. 分流：按业务优先级拆 Topic/Tag，关键消息优先消费。
3. 提升消费能力：减少慢 SQL、加索引、批量写、缓存预热。
4. 回溯根因：对照 lag 曲线与 DB/Redis/CPU 指标，定位瓶颈点。

### 选型说明
- 项目当前路径是“观测先行 + 业务限流 + 消费侧优化”，符合秒杀突发场景。

---

## 2.5 MQ 中消息过期失效了怎么办？

### 项目里现状
- RocketMQ 消息保留由 Broker 配置决定，项目层面通过 Outbox 持久化关键消息，降低“仅依赖 MQ 存活时间”的风险。

### 处理方案
- 短期：从 Outbox（pending/dead）重投恢复。
- 中期：调整 Broker 存储保留策略和磁盘容量阈值。
- 长期：关键业务做事件归档（消息审计表/对象存储）用于回放。

### 选型说明
- 对交易关键消息，项目更依赖“DB 可回放事实源（Outbox）”，而不是只依赖 MQ 保留时长。

---

## 2.6 如何保证数据一致性？事务消息如何实现？为什么不用事务消息而用 Outbox？

### 项目里怎么做
- 一致性主线：Redis Lua 预扣 + DB 事务建单 + 补偿回滚 + Outbox 可靠投递。
- 关键目标是“最终一致性可收敛”，并通过 pending/canceled/processing 避免补偿与消费竞态。

### RocketMQ 事务消息可行方案
- 半消息（prepare）先入 MQ。
- 执行本地事务（建单/库存）。
- 提交或回滚半消息。
- Broker 回查本地事务状态。

### 为什么当前选 Outbox
1. 业务状态可审计：Outbox 表天然可查可回放。
2. 中间件解耦：对 MQ 实现依赖较低，迁移成本更可控。
3. 工程复杂度更低：避免事务回查链路复杂性。

### 事务消息 vs Outbox
- 事务消息优点：理论上一致性窗口更小。
- 事务消息缺点：回查实现复杂、运维成本高。
- Outbox优点：落地快、易观测、易补偿、跨中间件复用。
- 当前项目阶段选 Outbox 是“收益/复杂度比”更优。

---

## 2.7 项目中用的哪种消息模型？消费模式是什么？

### 当前模型
- 模型：Topic 普通消息 + 延迟消息（30 分钟）。
- 消费模式：PushConsumer + Clustering（集群消费）。
- 语义：至少一次（At-Least-Once），业务侧通过幂等保障最终正确。

---

## 2.8 项目中有多少个 producer、broker、consumer、topic、queue？对应关系怎样？

### 按项目代码与当前本地环境
- Producer：1 个全局 Producer（group: `seckill_group`）。
- Consumer：2 个 PushConsumer 实例。
  - `seckill_tx_consumer_group` 消费 `seckill_tx_topic`
  - `seckill_order_timeout_consumer_group` 消费 `seckill_order_timeout_topic`
- 业务 Topic：2 个。
  - `seckill_tx_topic`（主流程建单消息）
  - `seckill_order_timeout_topic`（订单超时延迟消息）
- 系统 Topic（运行期可能自动出现）：
  - `%RETRY%seckill_tx_consumer_group`
  - `%RETRY%seckill_order_timeout_consumer_group`
  - `%DLQ%...`（仅在达到死信条件后出现）
- Broker：代码不写死数量；当前本地部署是单 NameServer + 单 Broker。
- Queue：代码未硬编码 queue 数，取决于 RocketMQ 中 Topic 的创建配置（可按压测与并发需求调整）。

### 对应关系
- 下单成功 -> 写 `seckill_tx_tasks` -> 调度器投递 `seckill_tx_topic` -> tx consumer 建单。
- 建单成功 -> 发延迟消息到 `seckill_order_timeout_topic` -> timeout consumer 到时检查并取消未支付订单。

## 2.9 如何保证消息的可靠投递与可靠消费？

### 可靠投递（发送侧）
- 关键动作先落 Outbox（DB 持久化）再投 MQ。
- 同步发送（拿到 Broker ACK）后才标记 sent。
- 失败重试（指数退避）+ 超阈值转 dead + 定时重放。

### 可靠消费（消费侧）
- 消费失败返回重试（`ConsumeRetryLater`）。
- 幂等消费：唯一键冲突视为已处理。
- 超时链路补偿：建单后延迟消息失败会写 `order_timeout_tasks` 继续补发。

### 仍需增强（可作为后续优化）
- 增加 dead/pending 积压告警阈值（Prometheus + Alertmanager）。
- 增加运营查询接口（按订单号追踪 Outbox/MQ/订单状态）。
- 对关键链路加 SLO：投递成功率、补偿时延、积压回落时间。
