## 3. 本地压测限制声明（报告必带）

1. 压测机与服务机同机，网络 RTT 与丢包开销几乎不体现。
2. 本报告不代表跨机、跨网段、线上生产真实吞吐上限。
3. 本报告主要用于：
   - 不同版本横向对比
   - 瓶颈定位（DB/Redis/MQ/应用）
   - 架构优化是否生效

## 4. 压测计划（本地最小闭环）

### 4.1 裸 DB 读基线（可选但建议）

- 脚本：`loadtest/benchmark_db_read.js`
- 目的：测 MySQL 纯读上限，作为缓存收益对照。
- 命令：

```powershell
k6 run loadtest/benchmark_db_read.js
```

### 4.2 应用层热点读

- 脚本：`loadtest/benchmark_app_read.js`
- 目的：测缓存加速后的读吞吐与尾延迟。
- 命令：

```powershell
k6 run loadtest/benchmark_app_read.js
```

### 4.3 全链路交易

- 脚本：`loadtest/benchmark_e2e_tps.js`
- 目的：测秒杀下单 -> 轮询 -> 支付闭环稳定性与 TPS。
- 命令：

```powershell
k6 run loadtest/benchmark_e2e_tps.js
```

### 4.4 长稳态（建议）

- 时长：30~60 分钟
- 目的：观察资源泄漏、积压增长、补偿任务是否收敛。

### 4.5 突发洪峰（建议）

- 时长：1~5 分钟
- 目的：观察瞬时高峰下退化行为（限流/错误码/尾延迟）。

## 5. 核心指标（必须采集）

### 5.1 吞吐与延迟

- RPS/QPS（HTTP 层）
- TPS（业务成功交易）
- P50/P95/P99（按接口拆分）

### 5.2 错误与稳定性

- HTTP 错误率（4xx/5xx）
- 业务错误码分布（42900/40001/50001 等）
- 超时率、重试次数、轮询超时数

### 5.3 资源与中间件

- CPU/内存/GC
- MySQL：慢查询数、连接数、锁等待
- Redis：命中率、命令耗时、连接池
- MQ：积压量、消费速率、重试、死信
- 补偿任务：扫描量、补偿成功率、滞留量

## 7. 口径判读（本地版）

1. 热点读：应用层 QPS/RPS 应 >= 裸 DB 读 QPS（同数据规模、同口径下）。
2. 闭环交易：TPS 通常低于 QPS/RPS，重点看 P95/P99 与错误率是否可控。
3. 长稳态：指标应稳定，不应出现持续恶化（内存增长、积压只增不减、错误率上升）。
