## 二、突发流量压测（seckill_burst_test.js）

### 2.1 关键指标

| 指标 | 值 | 评估 |
|------|---|------|
| 总请求数 | 169,328 | — |
| HTTP RPS | 4,230/s | 系统吞吐量上限（含被拒请求） |
| HTTP 200 成功数 | 419 (0.25%) | ⚠️ 极低 |
| 限流拦截数 | 168,909 (99.75%) | 🔴 **几乎全部被限流** |
| 秒杀成功数 | 411 | 仅 411/2000 库存被消耗 |
| P95 延迟 | 11ms | ✅ 极快 |
| P50 延迟 | 1.17ms | ✅ 极快 |

### 2.2 阈值检查

| 阈值 | 目标 | 实际 | 结果 |
|------|------|------|------|
| `biz_success_rate > 1%` | > 1% | 0.24% | ❌ FAIL |
| `p(95) < 500ms` | < 500ms | 11ms | ✅ PASS |
| `http_req_failed < 1%` | < 1% | 99.75% | ❌ FAIL |

### 2.3 根因分析

**🔴 核心问题：IP 级限流成为瓶颈**

所有 300 个 k6 虚拟用户从 `127.0.0.1` 发起请求，共享同一个 IP 限流器：

```
IPRateLimiter: 10 req/s, burst 20（middleware.go）
```

300 VU 产生 ~4,230 req/s，但 IP 限流器只放行 **~10 req/s**。  
40 秒内理论放行 ~400 个请求，实际 419 个（吻合）。  
其余 168,909 个请求全部收到 HTTP 429 + `code: 42900`。

**结论**：本次压测实际测试的是限流器的拦截能力（✅ 表现良好），而非秒杀核心链路的吞吐量。

## 三、完整链路压测（seckill_full_test.js）

### 3.1 关键指标

| 指标 | 值 | 评估 |
|------|---|------|
| 秒杀成功数 | 596 | 突破限流后的实际下单量 |
| 支付成功数 | 576 | ✅ 96.6% 支付成功 |
| 支付失败数 | 20 | ⚠️ 3.4% 支付失败 |
| 轮询超时数 | 0 | ✅ MQ 消费能力充足 |
| 售罄数 | 0 | 库存未耗尽（被限流挡住） |
| 重复购买数 | 21 | 正常（一人一单拦截生效） |
| 限流数 | 140,960 | 与 burst 测试一致 |
| 端到端 P95 | 516ms | ✅ 可接受（含轮询等待） |

### 3.2 深度分析

#### ✅ 乐观锁修复验证

| 对比项 | 修复前 | 修复后 |
|--------|--------|--------|
| DB 建单成功率 | ~270/2000 (13.5%) | 596/596 (100%) |
| 根因 | `WHERE version = ?` 乐观锁冲突，64 并发消费者竞争同一 version | 改为 `WHERE available_stock > 0`，InnoDB 行锁串行化 |

修复后 **所有通过限流的请求 100% 建单成功**，原乐观锁冲突问题已消除。

#### ✅ 支付链路验证

- 576/596 = **96.6%** 支付成功率
- 20 笔失败可能原因：
  - 轮询返回 SUCCESS 后到支付请求之间，MQ 延迟消息触发了超时取消（概率极低）
  - 并发窗口：同一用户的多个 VU 同时拿到 orderNo，但只有第一个支付成功
  - 网络抖动导致 HTTP 请求失败

#### ✅ MQ 消费能力

- 轮询超时数 = **0**，说明 MQ 消费者在 10s 轮询窗口内 100% 完成建单
- 消费并发度 64 goroutine（配置值）足以处理当前负载

#### ⚠️ 库存未耗尽

- 2000 库存只消耗了 596 个，剩余 1404
- 原因：IP 限流器只放行 ~10 req/s，大量用户无法进入秒杀核心流程
- 这不是 BUG，是限流策略在本地压测环境下的预期行为

#### 混合读写

| 指标 | 值 | 评估 |
|------|---|------|
| 混合读 HTTP 200 | 99% | ✅ 正常 |
| 混合写 HTTP 200 | 0.9% | 🔴 被限流 |

### 4.2 DB 性能分析

#### ✅ 写入性能优秀

DB 写入 P95 = 5ms，包含完整事务（INSERT order + INSERT order_item + UPDATE stock + INSERT log）。  
MySQL 连接池配置 maxOpenConns=500 绰绰有余。  
**DB 写入不是本系统的性能瓶颈。**

#### ⚠️ 读取延迟偏高

订单列表查询 P95 = 142ms，可能原因：
1. `COUNT(*)` 全表统计 + `ORDER BY created_at DESC` 分页，在数据量增大后性能衰减
2. 建议: 对 `orders` 表的 `(user_id, created_at)` 添加复合索引
3. 当前 GORM 自动建了 `idx_user_id` 索引，但未覆盖 `created_at` 排序

#### ⚠️ 轮询接口 11% 失败

轮询接口走鉴权路由（JWT + 用户级限流 2/s），500 VU 时部分用户超出限流阈值。  
这不是 DB 问题，而是限流策略在高并发读场景下的表现。

#### 🔴 写操作几乎全部被限流

与突发测试相同，IP 限流器将写操作 RPS 限制在 ~10/s。  
**无法通过现有脚本测试 DB 写入的真正瓶颈。**

## 六、后续建议

### 6.1 压测层面

**优先级 P0（阻塞性）**：调整 IP 限流策略以释放压测能力

现有 IP 限流器（10 req/s, burst 20）将所有本机压测请求限制在极低水平。  
建议以下方案之一：

| 方案 | 做法 | 优点 | 缺点 |
|------|------|------|------|
| A. 压测时临时放宽 IP 限流 | 改为 `10000/s burst 20000` | 最简单 | 需改代码 |
| B. 压测环境绕过 IP 限流 | 增加环境变量开关 | 不影响生产 | 需新增逻辑 |
| C. 使用多台压测机 | 不同 IP 发请求 | 最真实 | 需要资源 |

放开 IP 限流后，预期可达到的真实 RPS（受用户级限流 2/s 约束）：
- 10000 用户 × 2 req/s = 理论上限 **20,000 RPS**
- 实际受 MySQL/Redis/MQ 瓶颈约束，预期在 **3000-8000 RPS**

### 6.2 代码层面

1. **订单列表查询优化**：为 `orders` 表添加 `(user_id, created_at DESC)` 复合索引，将 P95 从 142ms 降至 <30ms
2. **轮询接口限流放宽**：`GET /seckill/result` 是无副作用的读操作，可提高用户级限流至 10/s
3. **DB 连接池监控**：当前 maxOpenConns=500，建议添加连接池使用率监控（ `SHOW STATUS LIKE 'Threads_connected'` ）

### 6.3 下一轮压测计划

| 轮次 | 目标 | 脚本 | 前置条件 |
|------|------|------|---------|
| Round 2 | 放开限流，测真实秒杀吞吐量 | `seckill_burst_test.js` | 放宽 IP 限流至 10000/s |
| Round 3 | 找性能拐点 | `seckill_stress_test.js` | 50→2000 VU 阶梯 |
| Round 4 | DB 读写瓶颈 | `seckill_db_test.js` | 放宽限流 + 大数据量 |
| Round 5 | 全链路端到端 | `seckill_full_test.js` | 验证 2000 库存全部消耗+支付 |
| Round 6 | 监控集成 | `seckill_monitor_test.js` | Prometheus + Grafana |
